# 要約（3行)
- Google Researchが、LLMエージェントの安全性向上を目的とした「VeriGuard」を提案（ACL 2026）。
- テキスト出力ではなく「行動を伴うコード生成」を検証する枠組みで、安全でない操作や権限逸脱を抑制する方向性を示す。
- プロンプトインジェクションやメモリ汚染といった新しい攻撃面に対し、従来のLLM安全対策では不十分である点を前提に設計されている。

## 1. 何が新しいか
- LLMの出力テキストではなく「エージェントが実行するコード」を安全性評価の中心に据えている点。
- Verified Code Generation（検証付きコード生成）という枠組みにより、実行前の安全性チェックを強調。
- 間接的プロンプトインジェクション（indirect prompt injection）やメモリ汚染（memory poisoning）を明示的な脅威モデルに含めている。
- 従来の毒性・バイアス対策から、操作・権限・環境相互作用にまで安全性スコープを拡張。
- LLMエージェント特有の「多段計画＋外部ツール利用」を前提とした安全設計。

## 2. 技術ポイント（エンジニア向け）
- 対象はツール・API・メモリにアクセスし行動を実行するLLMエージェント。
- リスク分類として、不正操作、プライバシー違反、タスク逸脱（misalignment）、攻撃耐性の欠如を明示。
- 攻撃ベクトルに indirect prompt injection と memory poisoning を含めている点が特徴的。
- 従来のLLM安全対策（毒性・バイアス制御）ではエージェントの行動リスクをカバーできないと指摘。
- 「コード生成」を仲介層として扱い、そのコードに対して何らかの検証プロセスを適用する構想。
- エージェントは知覚・計画・ツール利用・実行の一連のループを持つ前提で議論されている。
- 金融・医療・Web操作・ソフトウェア開発など、実環境との相互作用が前提のユースケースを想定。
- 安全性問題を「出力内容」ではなく「実行結果と副作用」の観点で再定義している。

## 3. 実装・運用への示唆
- エージェントに直接API呼び出しをさせるのではなく、一度コード生成に落とし込み検証層を挟むアーキテクチャが必要になる可能性がある。
- 権限管理やデータアクセス制御は、自然言語レベルではなくコードレベルでの静的／動的検証に移行する設計判断が求められる。
- プロンプトインジェクション対策として、外部入力をそのまま計画やコードに反映させない検証フローの導入が前提になる。
- メモリ機構を持つエージェントでは、保存情報の信頼性検証や汚染検出ロジックを別途設計する必要がある。
- タスク逸脱（misalignment）を防ぐには、ユーザー意図と生成コードの整合性チェックを組み込む必要があるが、具体手法は本文からは不確実。

## 4. 注意点・限界
- 本文が概要レベルにとどまり、VeriGuardの具体的な検証手法（形式手法かルールベースか等）は不明。
- 評価設定やベンチマーク、定量的な改善幅に関する情報が提示されていない。
- 実運用での性能オーバーヘッドやスケーラビリティに関する記述がない。
- どの粒度のコード生成（関数単位／スクリプト単位）を想定しているか不確実。
- 既存エージェントフレームワークとの統合方法や互換性は不明。

## 5. こんなチームに有効
- LLMエージェント基盤やAIエージェントプロダクトを開発するエンジニアリングチーム
- セキュリティ／アプリケーションセキュリティ（AppSec）担当
- API・権限管理・データガバナンス設計に関わるプラットフォームチーム

## 6. 5分で話せる説明
- まず前提として、LLMは単なるテキスト生成から「行動するエージェント」に進化している。
- その結果、安全性の問題もテキストの有害性ではなく、実際の操作やデータアクセスに広がっている。
- VeriGuardはこの点に対して、エージェントの行動をコードとして生成させ、そのコードを検証するというアプローチを取る。
- 特に、indirect prompt injectionやmemory poisoningのような新しい攻撃を想定している点が重要。
- 従来のフィルタリング中心の安全対策では不十分で、実行前検証という設計へのシフトが必要になる、という問題提起になっている。

## 7. 私の視点
- コード生成を介在させる設計は、監査可能性と再現性の観点で合理的だが、開発体験は確実に複雑化する。
- 実効性は「どこまで形式的に検証できるか」に依存し、軽量なルール検証では攻撃耐性が限定的になる可能性がある。
- エージェントの安全性は単一手法で解決する問題ではなく、VeriGuardはその一層（実行前検証）を担う位置づけに見える。
- メモリ汚染対策に踏み込んでいる点は現実的だが、運用上は検知と回復戦略の方がボトルネックになりそう。
